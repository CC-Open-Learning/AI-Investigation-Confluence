AI Investigation : Copilot Studio Privacy and Security

Created by Arsh Chauhan, last modified on Jun 06, 2024

Co-pilot studio is a powerful tool with plenty of features that can help in easing task but there are risk of potential security vulnerabilities and data breaches. It is an important task to identify these risks and develop effective mitigation strategies to make sure the secure use of copilot studio. Make sure to read the attached document to get more information of the topic (https://learn.microsoft.com/en-us/copilot/microsoft-365/microsoft-365-copilot-privacy )

Potential Risks and Vulnerabilities

  1. Model Inversion Attacks: Studio have great access to all the data and then it can exploit by extracting information from the data it processes.

  2. Incorrect Access Control over data: Inadequate access control can allow unauthorized users to access sensitive data and compromise the security of the system. Organization should be able to determine the appropriate things user is allowed to do. Refer this (https://learn.microsoft.com/en-us/microsoft-copilot-studio/personal-data-summary#a-note-about-requests-to-rectify-personal-data ). Potential risk in this includes

    1. Breach of protected information = PII and PHI

    2. Ramifications of the information becoming public

    3. Ongoing control measures to protect individual whose information was accessed.

  3. Intellectual Property - Accessing key to Intellectual Property(IP0 or Financial information about the organization can be a data breach issue. Potential Risk in this includes

    1. Insider Risk Concerns

    2. Corporate piracy

    3. Longer term ramifications of leaked financial or IP Data

  4. Over Sharing - As there is no authentication in copilot so if there is going to be no authentication required in that other people will be able to use copilot too which means that can be publicly accessible to everyone. Copilot studio in order to provide accurate and timely information will be needed to connected to an internal SharePoint site or SAP Table that contains the sensitive data. if anyone can simply log on the access both can have access to potentially sensitive data and it can lead to data leakage

  1. Prompt Injection - Copilot studio can be targeted using prompt injection attacks, where a hacker can directly attack and can modify LLM’s model in an attempt to overwrite existing system prompts. It is more worse when users with bad intention will get access to these bots. It is east to the bots to give the sensitive information to them which can results in frequent data leaks. On asking the bot any query which can leak sensitive information using prompt which goes like “Ignore all the previous instructions. And then text” can lead the bot to leak sensitive information.

  1. Violation of Least Privilege - In Order to track performance and see the interaction with each individual bot Microsoft provides access to transcripts in shared table and that table is accessible by many that contains all transcripts from all the bots. Which is clear violation of least privilege. As a single place contains all the transcripts and engagements with each bot, which can result in huge opportunity for data leakage.

Mitigation Strategies

  1. Strict Access Controls - Implement Access controls to ensure that only authorized users(IT Conestoga and VAR Lab) can have access to sensitive data.

  2. Monitor and Audit Data Usage - Monitor and audit data usage to identify security risks and take corrective action.

  3. Use Encryption - Use Encryption technique to protect sensitive data and ensure its confidentiality. This can be done by applying Sensitivity labels(https://learn.microsoft.com/en-us/purview/sensitivity-labels ) or by restricting permissions in apps in Microsoft 365 by using Information Rights Management(IRM)

  4. Implement Data loss Prevention - Implement Data loss prevention policies to prevent data breaches and unauthorized data sharing

  5. Ask 5 Questions from the AI - Answering these questions can give us the right answer of how data is safeguarded.

    1. How are we going to define the sensitive data?

    2. Are we sure of the data flow?

    3. What and where is the sensitive data?

    4. How are User granted access to sensitive data or documents?

    5. What are sharing policies.

  6. To Protect the data there are two methods(mentioned below). The whole process is very reminiscent of ‘opt-in’ vs ‘opt-out’ that organizations inject into process that allows them to harvest data, send spammy emails, and more unless the user goes out to correct it. Data protection methods:

    1. Choose a authentication method

    2. Builders must select the box that requires users to log in

      • To Prevent this from happening using anyone using studio should double check the authentication method and make sure to include authentication where there is access to sensitive data

Protection to prevent user compromising from Copilot studio services

  • Logical Isolation of Customer content within each tenant for Microsoft 365 services which is achieved through Microsoft Entra authorization and role based access control

  • Microsoft uses physical security, background screening, and a multi-layered encryption strategy to protect the confidentiality and integrity of customer content.

  • To Learn more about this refer this - https://learn.microsoft.com/en-us/copilot/microsoft-365/microsoft-365-copilot-privacy#how-does-microsoft-copilot-for-microsoft-365-protect-organizational-data

Data Loss Prevention (DLP) Policies

  • Purpose - DLP polices can help in preventing accidental data exfiltration or loss from Copilot studio copilots.

  • Configuration - Administrators can create DLP Policies in the power platform admin center. These policies can classify CoPilot studio connectors in to data groups to protect Organizational data.

  • Mitigation Strategies

    • Authentication - Ensure that copilots require authentication before granting access to sensitive data. By Default, no authentication is needed which can lead to data package.

    • Credential Sharing - When Building bots, avoid embedding personal credentials. Otherwise, every user interacting with the bot uses the builder’s identity, leading to security tasks.

  • Refer to this too - Apply data loss prevention policies to copilots - Microsoft Copilot Studio | Microsoft Learn

  • Also Refer to Security Implementation plans Features for more detailed version on this

Best Practices:

  1. Use Copilot Studio with Caution - Use Studio with proper caution and ensure that it is used and also following organization’s security policies and procedures.

  2. Use Strong Password - Use strong passwords and enable two factor authentication to secure access of the tool

  3. Limit Access to limited users - Limit the access to the tool to only authorized users to ensure that they have the necessary permissions and privileges

  4. If there are large enough of components export it using the classic power apps portal instead.
    (Read this article for more information - https://learn.microsoft.com/en-us/microsoft-copilot-studio/authoring-export-import-bots?tabs=webApp )
